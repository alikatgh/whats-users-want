What should we do

Three things to do

1

Give each recovery user one clear status

Reason, proof needed, owner, and next step travel with the UID — no more starting over each reply.

2

Treat repeat users as one case, not many tickets

When a user comes back, an owner picks it up where it left off instead of sending another isolated reply.

3

Keep moderation notes traceable to rows

No-ban mentions and similar context stay linked to the original ticket, so decisions can be reread later.

Counting method

Definitions

Analysis-ready records
Rows left after summary rows and helper columns are removed.
Tickets read by AI
Rows the local model read to extract the user’s goal.
Discovered wants
Goals the model found inside those tickets (e.g. access recovery).
Mapped records
Analysis-ready rows assigned to the nearest discovered want.
Timeline numbers
Per month, the count of rows mapped to a want. Partial months hidden by default.
Keyword numbers
Literal text matches. restore counts rows containing that exact word.
No-ban records
Rows containing no-ban, no ban, or no_ban.
Repeat users
UIDs that appear more than once, ordered by date.
